Challenges Identified

Technical problems

Real issues encountered during development and testing. Each problem is documented with root cause and current workaround or fix.

⚙️Claim Extraction Duplication
When compound sentences contain multiple connected clauses, the Groq extractor sometimes breaks them into overlapping claims or duplicates the extraction. This inflates claim count and lowers overall scores artificially.
Root Cause:
Input: "India is the world's largest democracy and gained independence in 1947." Extracted as: 1. "India is the world's largest democracy" 2. "India gained independence in 1947" 3. "India is the world's largest democracy and gained independence in 1947" ← duplicate
Symptom:
Scenario 3 returned 10 claims instead of 5. Score artificially lowered from expected 40–60% to 22%.
Impact:
Medium. Affects accuracy calculations since incorrect claims are weighted more if duplicated.
Workaround: Deduplicate extracted claims before scoring
🔀Fallback Verdict Resolver Forces Binary Outcome
When the primary verdict synthesis fails (insufficient evidence), the fallback resolver is called. It uses a prompt that explicitly forbids "unverifiable" verdict, forcing a binary choice between "correct" and "incorrect". This misclassifies future-tense or genuinely unprovable claims as "incorrect" with low confidence.
Root Cause:
// In fallback resolver - NEVER return "unverifiable". - If evidence is weak, choose "incorrect" with low confidence (20-45).
Symptom:
Scenario 5 marked "Mars colony by 2040" as incorrect (35% confidence) instead of unverifiable.
Impact:
Low for demo, but HIGH for production. Users expect future claims to be marked unverifiable, not wrong.
Fix: Remove NEVER constraint, allow unverifiable in fallback
📊Over-reliance on Web Snippets
The routing logic should prioritize Wikidata (structured, zero hallucination) for historical facts, but falls back to web snippets when Wikidata SPARQL returns empty results. This introduces lower-confidence sources for claims that should be easy to verify.
Root Cause:
Routing priority (current): 1. Try Wikidata (fast, reliable) 2. Try Wikipedia (free) 3. Fall through to Serper (web) 4. Fallback to snippets (unvetted) Issue: Step 1 returns empty in many cases.
Symptom:
Scenarios 1–2 show 4/5 claims sourced from "Web snippets" instead of structured data.
Impact:
Medium. Correct verdicts still reached, but source quality score (6–7) could be higher with better routing.
Optimization: Improve Wikidata SPARQL query generation for edge cases
⏱️Long Processing Time for Mixed Scenarios
Scenario 3 took 46.3s to process 5 input claims. With extraction duplication producing 10 claims, the semantic analysis step (scoring non-factual dimensions) runs 10 separate Groq calls with seed=42, causing timeouts.
Root Cause:
Dimension scoring loop (scoreNonFactualDimensions): for each claim: if contains non-factual dimensions: await groq.create(...) // 1 API call = ~2-4s 10 claims × 5 non-factual dims = 50 potential calls Actual: 46.3s for 10 claims (some cached).
Symptom:
Processing time jumped from 4–7s (scenarios 1–2) to 46s (scenario 3).
Impact:
High. User-facing timeout risk. Live demo needs <10s target.
Solution: Implement request batching and caching layer
🔍Non-deterministic Verdict Output (Fixed)
Initially, Groq API calls used random seed, causing the same claim to receive different verdicts on different runs. This broke the requirement for reproducibility.
Root Cause:
groq.chat.completions.create({ temperature: 0, // ✓ set to 0 // seed: missing ← No seed specified messages: [...] })
Solution Applied:
groq.chat.completions.create({ model: "llama-3.1-8b-instant", temperature: 0, seed: 42, // ← Added deterministic seed messages: [...] })
Status: FIXED – All Groq calls use seed=42
🎯Checkworthy Filter Too Aggressive Initially
Early versions filtered out valid claims like "The capital of India is New Delhi" because confidence heuristic scored them low. Threshold was set too high (60+).
Root Cause:
checkworthiness_score = (keyword_count + source_credibility) / 2 if checkworthiness_score < 60: return null // Skip this claim Issue: Legitimate geographic claims scored 45–55.
Solution Applied:
Lowered threshold from 60 to 30, combined with improved heuristics for entity and numeric claims.
Status: FIXED – Threshold reduced, validation passing
Mathematical Foundation

Scoring formulas

Transparent breakdown of how trust scores are calculated. All formulas use deterministic inputs (no randomness) to ensure reproducibility.

Data Accuracy (Foundation Metric)

Core metric used across all domain types. Combines correct/incorrect ratio with unverifiable penalty.

score = (correctCount ÷ totalClaims) × 10 unverifiableRate = unverifiableCount ÷ totalClaims score = score - (unverifiableRate × 3) score = Math.max(0, Math.min(10, score)) // Clamp: 0–10

Financial Reasoning (Domain-Specific)

For FINANCIAL content type. Weights data accuracy and numeric claims.

dataAccuracy = (correctCount ÷ totalClaims) × 10 numericClaims = count of claims with /\d+/ pattern boost = numericClaims > 0 ? 2 : -2 score = Math.round(dataAccuracy × 0.8 + boost) score = Math.max(0, Math.min(10, score))

Technical Precision (Domain-Specific)

For TECHNICAL content type. Checks for technical terminology.

technicalTerms = ["version", "api", "protocol", "server", "database", "code"] matches = technicalTerms.filter(t => text.toLowerCase().includes(t)).length score = Math.round((matches ÷ technicalTerms.length) × 10)

Regulatory Awareness (Domain-Specific)

For LEGAL and FINANCIAL content types. Detects regulatory language.

regulatoryKeywords = ["law", "regulation", "compliance", "statute", "mandatory"] matches = regulatoryKeywords.filter(k => text.toLowerCase().includes(k)).length score = Math.round((matches ÷ regulatoryKeywords.length) × 10)

Overall Trust Score (Weighted Composite)

Aggregate across all dimensions. Data Accuracy weighted 2×, others 1×.

totalWeight = 0 weightedSum = 0 forEach dimension: weight = (dimension.name === "Data Accuracy") ? 2 : 1 weightedSum += dimension.score × weight totalWeight += weight overallScore = Math.round((weightedSum ÷ totalWeight) × 10) / 10

Risk Level Classification

Deterministic bucketing based on overall score.

if (overallScore >= 6.0): return "LOW RISK — Generally Safe" else if (overallScore >= 4.0): return "MEDIUM RISK — Review Recommended" else: return "HIGH RISK — Do Not Execute"

Semantic Dimension Scoring (Non-Factual)

For dimensions like "Human Oversight Compliance", "Action Safety Gating", etc. Uses Groq LLM with seed=42.

// Groq-rated scale 0–10: // 0–2: None or implicit // 3–4: Minimal, buried in text // 5–6: Moderate, somewhat clear // 7–8: Explicit, well-documented // 9–10: Unmistakable, prominent score = clamp(groq_output, 0, 10)
System Design

Pipeline architecture

Six-stage verification pipeline from raw text to trust report. Each stage is independently testable and fault-tolerant with fallback paths.

STAGE 1 · INPUT INGESTION
Text (≤10K chars) → Auto-detect content type → Store for highlighting
✓ Supports GENERAL, FINANCIAL, MEDICAL, LEGAL, TECHNICAL
STAGE 2 · CLAIM EXTRACTION
Groq: "Extract verifiable factual claims only"
Filter: opinions, rhetorical, meta-linguistic
Issue: duplication in compound sentences (see problems)
STAGE 3 · CHECKWORTHINESS SCORING
Llama score 0–100: is this claim worth verifying?
Filter: threshold 30+ → reduces API calls by 40–60%
STAGE 4 · SMART ROUTING (NOVEL)
Route each claim to optimal source:
- HISTORICAL → Wikidata
- RECENT_EVENT → Serper
- PERSON/PLACE → Wikipedia
- POLITICAL → FactCheck API
STAGE 5 · PARALLEL SOURCE FETCH
Promise.all([wikidata, wiki, serper, factcheck])
Timeout: 8s per claim
STAGE 6 · VERDICT SYNTHESIS
Groq: "Synthesize verdict from evidence"
Fallback: force binary if unsure
✓ Deterministic (seed=42, no randomness)
